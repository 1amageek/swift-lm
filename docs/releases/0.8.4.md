# swift-lm 0.8.4

`0.8.4` is a startup-cost and steady-state inference release. It separates the
synchronous prewarm path from inference timing so callers can attribute every
millisecond between `ModelBundleLoader.load(...)` and the first generated
token, and it eliminates per-generate `MetalInferenceModel` re-initialization
by reusing the prewarmed prototype context across sequential calls.

## What 0.8.4 Changes

### Phase-separated timing logs

Loader, STAF, compiler, pipeline, residency, and model init all now emit
prefixed `[Prewarm/...]` lines so the cost of `load(...)` is fully attributed.
Streaming generation now emits `[Inference]` summaries for prefill, decode,
and total per call.

```
[Prewarm/Loader] inspect: 0.004s
[Prewarm/STAF]   metadata: 0.069s
[Prewarm/STAF]   cache hit  (validated in 0.001s)
[Prewarm/STAF]   mmap: 0.000s
[Prewarm/Loader] tokenizer: 0.448s
[Prewarm/Compiler] decode plan build: 0.004s
[Prewarm/Pipeline] decode base PSO loop: 0.006s (0 hits, 26 misses)
[Prewarm/Residency] runtime lease: 0.017s (21 buffers)
[Prewarm/Residency] weight lease: 0.868s (1 buffers)
[Prewarm/Loader] MetalInferenceModel.init: 0.901s
[Prewarm] TOTAL: 1.405s — lfm2 ready
[Inference] prefill: 0.023s (2 tokens, 86 tok/s) — TTFT
[Inference] decode:  0.474s (3 tokens, 6.3 tok/s)
[Inference] TOTAL:   0.497s (4 visible tokens, 8.0 tok/s aggregate)
```

`load(...)` already pays the full prewarm cost synchronously (STAF mmap,
tokenizer load, IR resolve, kernel compile, MTLResidencySet promotion, GPU
sync). Nothing has changed about *when* prewarm runs; the new logs only make
it observable. There is still no separate `prewarm` step to invoke.

### Prototype context reuse

`LanguageModelContainer.generate(_:parameters:)` and
`generate(from:parameters:)` previously cloned a fresh `LanguageModelContext`
per call, which re-initialized the prototype's
`MetalInferenceModel`-equivalent state at the cost of buffer zeroing and
residency promotion on the hot path.

The container now leases the prewarmed prototype context when it is idle and
only clears decode state (KV cache, scratch). Concurrent calls fall back to
cloning a fresh isolated context so generations remain non-interfering. Lease
release is driven by the `AsyncStream` lifecycle (consumer drain or
`onTermination`) and guarded by an idempotent `OnceBox` so the prototype is
released exactly once per generation.

For most applications — sequential request/response — back-to-back
`container.generate(...)` calls now skip per-call model re-init entirely and
reuse the wired-in weight residency.

Callers that need guaranteed parallelism should keep holding their own
`LanguageModelContext` instances; behavior there is unchanged.

### Pipeline cache correctness fix

The process-wide `MTLComputePipelineState` cache in `MetalPipelineCompiler`
was previously keyed by Metal function name only. Two unrelated containers
that emit functions with the same name but different MSL bodies (for
example, a small synthetic test container with `hiddenSize=64` and a real
production container with `hiddenSize=2048`) would silently share the wrong
pipeline. The GPU then ran kernel code compiled for one container's
dimensions and constants against another container's buffers, producing
deterministic decode garbage that surfaced as token-level corruption rather
than a crash.

The cache is now keyed by `(SHA256(MSL source), function name)`, so
pipelines are reused only when the source is byte-identical. This restores
correctness when multiple containers are alive in the same process and
removes a subtle source of cross-test interference. There is no API change
and no behavior change for single-container processes.

## Correctness

Validated before release with focused real-bundle E2E tests across every
thinking-capable family:

- LFM2.5 separate-thinking E2E emits non-empty reasoning **and** non-empty
  visible answer; HuggingFace token prefix parity holds.
- LFM2.5 hidden-thinking E2E emits visible answer with no `<think>` /
  `</think>` leakage.
- `Container.generate(...)` reuses the prewarmed prototype across two
  sequential calls (new test).
- Qwen3.5 prompt-opened thinking E2E and public `generate` E2E continue to
  pass.
- Gemma4 thinking-separate, hidden-reasoning, and public `generate` E2E
  continue to pass with no thought-channel leakage.

```bash
xcodebuild test-without-building -scheme swift-lm-Package -destination 'platform=macOS' \
  -only-testing:'SwiftLMTests/ReleaseSmokeOutputTests' \
  -only-testing:'SwiftLMTests/QwenVisionRealBundleTextTests' \
  -only-testing:'SwiftLMTests/Gemma4RuntimeTests' \
  -parallel-testing-enabled NO
```

## Notes

- No public `SwiftLM` API migration is required.
- The `[Prewarm/...]` and `[Inference]` log lines are written via `print`
  (stdout). Hosts that route logs elsewhere can filter on the prefix.

## Installation

```swift
dependencies: [
    .package(url: "https://github.com/1amageek/swift-lm.git", from: "0.8.4")
]
```
