# swift-lm 0.8.3

`0.8.3` is a Gemma4 chat-rendering correctness release. It removes the
hardcoded Swift implementation that bypassed the bundle's
`chat_template.jinja` and prevented Gemma4 thinking mode from activating, and
unifies all chat rendering through a single Jinja path.

## What 0.8.3 Changes

### Gemma4 thinking activation

Previously, `LanguageModelContext` rendered Gemma4 prompts with a hand-written
Swift template that ignored `enable_thinking`, so Gemma4 thinking bundles never
emitted reasoning even when `PromptPreparationOptions.isThinkingEnabled` was
`true`. The Swift template path is removed. Gemma4 prompts now render through
the same Jinja code path used for every other model:

1. The bundle's `chat_template.jinja` is loaded if present.
2. Otherwise, the `chat_template` field of `tokenizer_config.json` is used.
3. Otherwise, for Gemma4 model types only, a synthesized fallback template is
   provided by `Gemma4DefaultChatTemplate`. The fallback mirrors the upstream
   prompt shape, including `<|think|>` system-header injection when
   `enable_thinking` is truthy.

`bos_token` is no longer hardcoded in Swift; it is supplied to the template as
a Jinja variable, matching the upstream `chat_template.jinja` exactly.

### Stricter fallback for missing chat templates

The non-template fallback in `LanguageModelContext.applyChatTemplate` no longer
emits Qwen-specific vision markers (`<|vision_start|>...<|vision_end|>`) for
arbitrary models. When a bundle has no chat template and the input contains
images or videos, `prepare(_:)` now throws
`LanguageModelContextError.unsupportedInputForModel` instead of silently
producing model-specific tokens that the target model does not understand.

Text-only fallback rendering (role-prefixed transcript) is unchanged.

## Correctness

Validated before release with focused real-bundle E2E tests:

- Gemma4 thinking-enabled prompt renders the `<|think|>` system header.
- Gemma4 separate reasoning E2E emits non-empty reasoning **and** non-empty
  visible answer with no channel-tag leakage.
- Gemma4 hidden reasoning E2E emits visible answer with zero reasoning chunks
  and no `<|think|>` / `<|channel>thought` / `<channel|>` leakage.
- LFM separate and hidden thinking E2E continue to pass.
- Qwen3.5 prompt-opened thinking E2E continues to pass.

```bash
xcodebuild test-without-building -scheme swift-lm-Package -destination 'platform=macOS' \
  -only-testing:'SwiftLMTests/Gemma4RuntimeTests/realGemma4PublicGenerateThinkingSeparateE2E()' \
  -only-testing:'SwiftLMTests/Gemma4RuntimeTests/realGemma4PublicGenerateThinkingHiddenE2E()' \
  -only-testing:'SwiftLMTests/Gemma4RuntimeTests/realGemma4ThinkingPromptRendersTrigger()' \
  -only-testing:'SwiftLMTests/ReleaseSmokeOutputTests' \
  -only-testing:'SwiftLMTests/QwenVisionRealBundleTextTests' \
  -parallel-testing-enabled NO
```

## Notes

- No public `SwiftLM` API migration is required.
- Bundles with neither a chat template nor a tokenizer-side chat template will
  now reject multimodal input explicitly instead of producing wrong tokens.
  This affects synthetic test fixtures only; all known production bundles ship
  a chat template.
