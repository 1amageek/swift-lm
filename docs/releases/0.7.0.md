# swift-lm 0.7.0

`0.7.0` is a refactor-focused minor release that promotes `QuantizationFormat`
to the single source of truth for weight and KV-cache dispatch, fixes two
latent correctness bugs in the Q4 decode path, and eliminates silent fallback
in FlashAttention's KV-cache read/write kernels. The protocol-driven dispatch
paves the way for Q2/Q3/Q5/Q6 support — their format structs and registry
entries are in this release as scaffolding, with GEMV/GEMM integration to
follow in 0.7.x.

## What 0.7.0 Changes

### QuantizationFormat as the single source of truth

Previously, kernel dispatch for quantized weights and KV caches made
decisions via scattered switches over `WeightFormat` / scheme ID. `0.7.0`
consolidates all such decisions behind the `QuantizationFormat` protocol:

- **Decode GEMV**: Q4/Q8 migrated to a unified, format-driven generator.
  New formats only need to implement the protocol — no compiler changes
  required.
- **Prefill GEMM**: quantized GEMM dispatch now goes through the same
  protocol, collapsing a second set of ad-hoc switches.
- **KV cache sizing**: byte sizing derived from `QuantizationFormat`
  instead of a hardcoded table, so new KV schemes plug in uniformly.
- **Classification hooks**: the protocol now exposes `isAligned`,
  `isSupportedForKVCache`, and MSL generation helpers, removing the
  need for compiler-side knowledge of individual formats.

### KV cache silent-fallback elimination

FlashAttention's MSL read/write kernels previously fell back to a
hardcoded Q8 path when presented with an unsupported scheme, masking
misconfigurations at runtime:

- Write path unified behind `write_kv_quantized(scheme, ...)` that
  routes to `write_kv_quantized_q4` / `_q8` based on
  `kv_quant_bits(scheme)`. Rotor schemes resolve to their base scheme.
- Read path returns `NAN` for unsupported schemes instead of silently
  reading Q8 bytes.
- Swift-side `KVCacheSpecification.init` now has a precondition
  backed by `isSupportedForKVCache`, catching unsupported schemes at
  compile-plan construction rather than inside a Metal kernel.
- A new `KVCacheKernelSupportContractTests` suite locks the supported
  set and verifies rotor ↔ base agreement.

### Q4/Q8 decode GEMV grid mismatch fix

The Q4/Q8 decode GEMV kernels had a grid shape that did not match the
threadgroup size computed from `tptg.x`, producing wrong output on some
row counts. `0.7.0` derives `rowsPerThreadgroup` directly from `tptg.x`,
and adds a dedicated Q4 prefill GEMM path that doesn't share GEMV's
block layout assumptions.

### Q4 argmax-match criterion replaced with token-diversity

The previous Q4 correctness test required Q4 argmax tokens to match BF16
argmax tokens — a threshold that is not physically reachable for a
quantized model at a fixed temperature. `0.7.0` replaces it with a
token-diversity criterion (verifying the model doesn't collapse to a
single token), which correctly distinguishes "Q4 generates plausible
text" from "Q4 is broken".

### Q2 / Q3 / Q5 / Q6 / Q8G128 scaffolding

This release adds the format structs, registry entries, bit-level
dequant tests, and a unified GEMV generator skeleton for Q2, Q3, Q5,
Q6, and Q8G128. They are **scaffolding only** — not yet wired into the
end-to-end GEMV/GEMM dispatch. Loading a real MLX bundle in one of
these formats is not supported in 0.7.0. Full support will land in
0.7.x.

### Test refinements

- 13 MLX→STAF→GEMV round-trip tests covering every currently-supported
  format.
- Regression tests for sandwich-norm 4-way and LayerScale 5-way fusion.
- `RotorQuantRealBundleTests` rewritten: the previous assertion required
  Gemma4-E2B's very first token to equal `"Tokyo"`, which is unrealistic
  for an instruct-tuned model whose natural answer is `"The capital of
  Japan is Tokyo."`. The new assertion generates a 32-token window and
  verifies `"Tokyo"` appears anywhere in the answer.

## Correctness

- All Q4/Q8 GEMV contract tests pass.
- `KVCacheKernelSupportContractTests` (5 tests) pass, confirming the
  refactor is contract-preserving.
- `RotorQuantRealBundleTests` (5 suites: FP16 / RotorQ8-K / RotorQ8-V /
  RotorQ8 / RotorQ4) pass end-to-end on Gemma4-E2B with identical model
  output `"The capital of Japan is Tokyo."`.

## Upgrade notes

- `WeightFormat` remains as a type alias / migration surface; consumers
  should not observe API changes.
- If you constructed `KVCacheSpecification` with a non-KV-supported
  scheme (Q2/Q3/Q5/Q6 — none were actually usable at runtime), you will
  now hit an explicit precondition at plan-construction time. This is
  intentional.
- No changes to the public `SwiftLM` / `ModelBundleLoader` API.
